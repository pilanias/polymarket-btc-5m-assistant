# Milestone v1.1: Supabase Persistence

**Status:** SHIPPED 2026-02-24
**Phases:** 6
**Total Plans:** 1

## Overview

Replace ephemeral SQLite (better-sqlite3) with Supabase (hosted PostgreSQL) so trade history survives DigitalOcean App Platform deploys permanently. DigitalOcean Amsterdam does not support persistent volumes, causing SQLite data loss on every deploy.

## Milestone Summary

**Key Decisions:**

- Decision: Supabase REST client over direct PostgreSQL (Rationale: DO App Platform friendly, no connection pooling needed)
- Decision: service_role key not anon key (Rationale: Full database CRUD access required)
- Decision: Fire-and-forget trade sync pattern preserved (Rationale: PaperExecutor stays sync, Supabase writes async in background)
- Decision: globalThis singleton for cross-module store access (Rationale: ESM doesn't share module state across dynamic imports)
- Decision: Upsert for trade inserts (Rationale: Idempotent, safe for retry/migration)

**Issues Resolved:**

- SQLite data loss on every DigitalOcean deploy (core problem)
- 5 pre-existing test failures (import mismatches, key names, field names)
- Supabase anon vs service_role key confusion during setup

**Issues Deferred:**

- Live trading writes not yet tested against Supabase (paper mode only verified)
- No Supabase Row Level Security policies (service_role bypasses RLS)

**Technical Debt Incurred:**

- Phase 6 was implemented without formal GSD plan/execute workflow (no PLAN.md or SUMMARY.md)
- tradeStore.test.js skips all tests when better-sqlite3 unavailable (no Supabase-specific tests yet)
